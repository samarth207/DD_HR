New Test Case Guide

Simple instructions for adding and running new automated test cases.

Back to Easy Mode Open Technical Report

Quick Steps (No Coding)

  1. Open http://localhost:3000/testing-easy.html
  2. Fill Create New Feature Test form
  3. Click Generate Test File
  4. Run npm run test:e2e:report
  5. Check results on Easy page and Report page

What To Check For Every New Test

  1. API response is correct
  2. Database values are correct
  3. Business rule is correct
  4. Expected vs actual values are recorded
  5. Cleanup rollback is successful

What To Enter In The Form (Simple Explanation)

Module Name

Business area label like Leave, Attendance, Sales, Salary, Employee.

HTTP Method

API action type: POST create, GET read, PUT/PATCH update, DELETE remove.

Expected Status Code

Expected response code from your API. Common values: 200, 201, 400, 401, 403, 404.

API Endpoint

Endpoint is custom based on your feature. It is not fixed. Use real routes like /api/leaves, /api/sales, /api/attendance/settings. Do not keep /api/example.

JSON Payload

Input data sent to API. Must be valid JSON. Example:

{
  "employeeId": 100001,
  "leaveType": "Paid Leave",
  "startDate": "2026-07-20",
  "endDate": "2026-07-20",
  "status": "pending"
}

Are API Endpoints Fixed Or Custom?

Custom. You should select the endpoint for the feature you are testing.

  • If you test leave creation, use /api/leaves
  • If you test sales update, use /api/sales
  • If you test attendance settings, use /api/attendance/settings

So the endpoint changes feature-to-feature.

Expected Status Code (How To Choose)

  • 200: successful request
  • 201: successful create
  • 400: request rejected by validation/business rule
  • 401: not logged in / invalid auth
  • 403: not allowed
  • 404: resource/route not found

Choose the code your feature should return when behavior is correct.

Example (Leave Feature)

Case: Employee applies leave on a holiday and system should reject it.

  • Module Name: Leave
  • HTTP Method: POST
  • API Endpoint: /api/leaves
  • Expected Status Code: 400 (rejection is expected)
  • JSON Payload:
{
  "employeeId": 100001,
  "leaveType": "Paid Leave",
  "startDate": "2026-07-15",
  "endDate": "2026-07-15",
  "status": "pending"
}

If API returns 400, this test is considered passed for this scenario.

What Each File Does

tests/e2e/automation.e2e.test.js

Main end-to-end workflows.

tests/e2e/generated/*.test.js

New tests generated from form/CLI.

testing/factories/seed-factory.js

Creates sample data for tests.

testing/assertions/validation-engine.js

Reusable assertions and checks.

testing/framework/cleanup-manager.js

Cleans up test data after each case.

routes/testing.js

Dashboard/report APIs and case generation API.

Troubleshooting

  • If dashboard has no records, run npm run test:e2e:report (not just npm run test:e2e).
  • If server fails from workspace root, run with prefix: npm --prefix "c:\Users\samth\Desktop\DD\HR\server" start.
  • If generated test fails, verify endpoint starts with /api/ and payload is valid JSON.
